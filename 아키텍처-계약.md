# ALPEXA — 아키텍처 계약 (돈은 어떻게 움직이는가)

> **이 문서는 "탑다운 설계도"다.** 위에서 규칙 하나를 정하고, 아래로 내려가며 모든 돈 경로를
> 그 규칙에 맞춘다. 새 코드를 짜기 전·리뷰할 때 **이 계약을 먼저 본다.** 계약을 어기는 코드는
> 머지 금지. (CLAUDE.md #5의 구체화 — 같이 읽는다.)

---

## 0. 한 줄 계약 (전부 여기서 파생된다)

> **돈은 서버만 주인이다. 브라우저(앱)는 보여주기만 한다.**
> 모든 돈 이동은 **종류별로 정확히 하나의 서버 RPC**로 일어나고, **멱등(ref)** 하며,
> 클라이언트는 돈 테이블에 **절대 쓰지 않는다.**

은행 비유: 서버=금고+은행원(돈을 만질 수 있는 유일한 존재), 앱=CCTV 화면(구경만).

---

## 1. 돈의 진실은 어디에 사는가 (소유권 표)

| 테이블 | 무엇 | 누가 쓸 수 있나 | 클라가 쓰면? |
|---|---|---|---|
| `accounts.balance` | 현금 잔고 | **서버만** (`trg_apply_ledger`가 ledger에서 파생) | ❌ 금지 |
| `ledger` | 모든 돈 이동 기록(멱등 ref) | **서버 RPC만** | ❌ 금지 (머니프린팅) |
| `positions` (sports `kind=bet`) | 베팅(스테이크 차감됨) | **`place_bet` RPC만** | ❌ 금지 (#13 공짜베팅) |
| `crypto_holdings` | 크립토 보유(진짜 코인) | **`crypto_trade`/`swap`/`stake` RPC만** | ❌ 금지 |
| `crypto_stakes` | 스테이킹 | **`stake_crypto`/`unstake_crypto` RPC만** | ❌ 금지 |
| `positions` (fx/crypto 표시 미러) | 매니저 대시보드용 표시 | 서버가 진실(`fx_open`/`crypto_holdings`) | ⚠️ 표시 미러는 서버 진실을 읽어 만들어야 함(아래 #4) |

> **불변식:** `accounts.balance == 오프닝 + Σ(ledger)`. 모든 열린 베팅엔 대응하는 `betstake-<id>` 차감이 있다.

## 2. 돈을 옮기는 문 (RPC = 유일한 통로, 종류별 하나)

| 돈 이동 | 문(RPC) | 멱등 ref | 차감/지급 위치 |
|---|---|---|---|
| 베팅 | `place_bet` | `betstake-<id>` | 서버: ledger −stake + positions |
| 캐시아웃 | `cash_out` | `betpay-<id>` | 서버 |
| 앱간 이체 | `app_transfer` | `xfer-...` | 서버 (원자적 양다리) |
| 출금 보류 | `withdraw_hold` | `wdhold-<id>` | 서버 |
| 크립토 매매 | `crypto_trade` | (트레이드 ref) | 서버: crypto_holdings |
| 크립토 스왑 | `swap_crypto` | | 서버 |
| 스테이킹/해제 | `stake_crypto`/`unstake_crypto` | | 서버: crypto_stakes |
| FX 진입/청산 | `fx_open`/`fx_close` | local_id | 서버 (청산 시 P&L) |
| 정산(스포츠) | Edge `sports-settle` | positions 삭제 선점 | 서버 |

**읽기 전용(돈 안 움직임):** `withdrawable_for`, `acct_for_email`, `login_email`, `pullBalances`.

## 3. 클라이언트가 할 수 있는 것 / 없는 것

**할 수 있는 것 (구경):**
- 매 로드마다 서버에서 잔고·보유·베팅을 **fetch**해서 **표시**.
- 잔고는 **메모리에만** 보관(`window.__srvBal`, React state). **localStorage 금지.**
- 돈 이동은 위 RPC **호출만**(요청). 결과는 서버가 결정.

**절대 금지:**
- 돈을 `localStorage`에 "진실"로 저장 (#9 — 계정간 누수원).
- `ledger`/`accounts.balance`/`positions(bet)`/`crypto_holdings`에 직접 쓰기.
- 같은 돈 이동을 두 경로로 (예: place_bet + 클라 insert) (#13).
- 표시 정체성(`alpexa.me`)과 다른 세션으로 거래 (#14 — assertIdentity가 막음).

## 4. 지킴이 (규칙을 "강제"하는 도구)

1. **RLS** — 돈 테이블 쓰기는 서버(SECURITY DEFINER RPC)만. 클라 직접쓰기는 DB가 거부.
2. **`tests/diagnose.js`** (poka-yoke 정적 스캔) — 클라 ledger insert(A2)·balance update(A6)·돈 localStorage(LS9)·**스포츠 베팅 클라 insert(BET)** 등을 정적으로 차단.
3. **`tests/*.test.js`** — 행위 회귀(예: `bet-server-only`, `identity-guard`, `settlement-server-only`).
4. **게이트:** `node tests/verify.js` 🟢 아니면 "완료" 금지.
5. **정체성 가드:** `AlpexaSync.assertIdentity()` — 세션≠표시계정이면 재로그인 강제.

> **새 버그를 칠 때:** `결함-로그.md`에 클래스 한 줄 + (행위면) `*.test.js` / (패턴이면) `diagnose.js` 체크 추가.

## 6. 작업 순서 (이 계약을 100%로)

1. **A 확인** — `force_opening_balance` 트리거가 오프닝을 강제하는지 SQL로 확인. 안 하면 닫기.
2. **B** — 매니저가 `crypto_holdings`/`crypto_stakes` 직접 읽기 + 크립토 `push()` 미러 폐기.
3. **C/D** — FX positions 미러 통일 + `fx_open` 마진 서버검사.
4. **E** — 크론 fail-closed 배포.

각 단계: READ-BEFORE-WRITE 표 → 불변식 → 승인 → 수정 → `verify.js` 🟢 → poka-yoke.
