### 1-5. Supabase Realtime — ⚠️ 핵심 격차 (대시보드가 앱보다 빨라지는 지점)
- **crypto-live.html에는 prices Realtime 구독이 없다.** 채널은 `crypto_holdings`(13852-13854)·`crypto_stakes`(13929-13931) 뿐 — CLAUDE.md 백로그 "Realtime 패턴 crypto-live.html 확산(잔여)"과 일치.
- **작동 실증 패턴 = trading.html:4252-4261**: `channel('prices-live').on('postgres_changes', {table:'prices'}, m => 단일행 머지)` — 행 단위 MERGE(다른 심볼 절대 안 지움), 폴링은 폴백으로 유지, 두 경로 멱등.
- publication SQL 근거: 저장소 내 명시는 `fx_pending`(fx_pending.sql:68)뿐이고 prices는 대시보드에서 직접 추가된 것으로 추정 — **대시보드 구현 시 구독 후 이벤트 실수신을 눈으로 확인**하고, 안 오면 `alter publication supabase_realtime add table public.prices` 배포(사용자 실행).

### 1-6. 차트 이력
- ⚠️ `alpexaFetchChart`가 **2번 선언**됨: Binance klines 판(:587-600, `api.binance.com/api/v3/klines`)이 **뒤에 선언된 CoinGecko market_chart 판(:14447-14469)에 덮여** 런타임 유효본은 CoinGecko(1m/5m는 합성 유지). 대시보드는 이 모호함을 복제하지 말고 **Binance klines 클라 직결**(CLAUDE.md 2026-07-13 라이브 검증: M1 실구조)을 명시적으로 단일 채택.
- 미니 스파크라인: 앱은 CoinGecko `sparkline_in_7d`(:14436-14438). 서버 캐시 `sparklines` 테이블은 §4-3 — 대시보드 와치리스트는 그쪽이 즉시·무한도.

## 4. Edge 캐시/보조

### 4-1. `feed` — 1초 캐시 읽기 프록시
- 모듈 레벨 `TTLCache(1000)` + single-flight: N명 폴링 → DB 초당 1회. `?t=prices`(또는 POST body) → `symbol,mid,spr_pts` 전행. HTTP `Cache-Control: max-age=1`로 CDN 겹층. — feed/index.ts:30, 40-48, 25
- 클라 폴백 순서 (계약): **feed 성공 → 사용 / null → prices 직쿼리** — 대시보드도 동일 순서. — crypto-live.html:14410-14415

### 4-2. `crypto-price` — §1-2 (prices 필자, 3초 크론)

### 4-3. `sparkline-cache` — 워치리스트 미니차트 서버 캐시
- `sparklines.id='all'` = `{sym:[≤48 closes]}` (크립토=미러 klines 15m, FX/주식=Yahoo), `id='strip'` = 랜딩 마켓 스트립. 크론 10-15분, CRON_SECRET fail-closed. — sparkline-cache/index.ts:18, 61-69, 146-153, 115-117
- ⚠️ crypto-live는 이 테이블을 **읽지 않고** CoinGecko sparkline을 씀(:14436-14438) — 대시보드는 `sparklines` 1행 읽기 채택(즉시 표시·프로바이더 무접촉).

### 4-4. `transak-widget` — 카드입금 온램프
- 서버가 api-secret으로 refresh-token → session → `{ok,widgetUrl}` (~5분 유효, 1회용). 클라는 amount/walletAddress/email 등만 전달. — transak-widget/index.ts:29-42, 62-85; 호출 crypto-live.html:5195-5205

### 4-5. `withdraw-otp` — 출금 이메일 OTP 발송
- JWT에서 신원 파생(클라 이메일 불신), 6자리 코드의 **SHA-256 해시만** `withdraw_otp`에 저장(10분 만료), Resend 발송, 응답 `{ok,sentTo:마스킹}`. — withdraw-otp/index.ts:56-92; 클라 흐름 crypto-live.html:12996-13006

## 5. 대시보드 데이터 레이어 설계 (sports-dashboard 스토어 패턴 이식)

### 5-1. 스토어 골격 — gamesStore/moneyStore 문법 그대로
sports-dashboard.html의 검증된 패턴: `{ state, subs:new Set(), sub(f){…return unsub}, emit(), async pull(), start(){pull + setInterval} }` — gamesStore :585-610, moneyStore :969-1075. 렌더는 전부 `store.sub(render)` 구독으로, 돈 변동 반영은 **오직 `pull()` 재조회**(클라 델타 금지 — :1284, 1500, 1519 주석이 원칙 실증).

### 5-2. 제안 스토어 4개
```
pricesStore   { map:{sym:{price,chg,ts,src}}, live:bool }
holdingsStore { rows:{asset:qty}, stakes:{asset|period:{usd,principal,period,since}}, ts }
balancesStore { crypto,fx,sports, fxEquity, withdrawable, ts }
txnsStore     { trades:[crypto_trades rows], finTxns:[requests rows] }
```

### 5-3. 구독/폴백 우선순위 (신선한 쪽이 이김, 전 경로 멱등)
| 스토어 | ① 최우선 | ② 준실시간 | ③ 폴백 | 주기/근거 |
|---|---|---|---|---|
| prices | Binance WS 직결(+Coinbase 6s 폴백) — crypto-live :521-582 이식, 80ms 스로틀 | Realtime `channel('prices-…')` prices 단일행 머지 — trading.html:4252-4261 패턴 | feed Edge(1s 캐시) → prices 직쿼리, 8s 폴 | WS ms / RT ~0.5s / 폴 8s |
| holdings·stakes | Realtime acct 필터 — crypto-live:13852-13854, 13929-13931 | — | 8s 폴 + visibilitychange, **7s busy 가드** (:13815) | RPC 성공 직후 즉시 재adopt (:13443-13444 패턴) |
| balances | RPC 직후 `pullBalances()` 즉시 (:13447) | — | 10s 폴 (:14590-14592) | alpexa-sync.js:219-244 |
| txns | RPC 성공 직후 pull | — | 열림 시 + 30s | crypto_trades(RLS owner) + requests |

컨트롤류(halt 5s :14645-14648, pricing_marks 5s :14638-14642)는 앱 주기 그대로 락스텝.

### 5-4. 신선도 게이트 (스포츠 oddsStale 문법의 크립토 대응)
- 심볼별 `ts` 스탬프(WS·RT·폴 공통). **stale > 15s** → 시세 dim + "Delayed" 뱃지 + feedDot stale (sports-dashboard:601, 605, 608-609 문법). **stale > 120s** → 해당 심볼 **거래 버튼 잠금** — 서버 `crypto_trade`의 120s 거절(crypto_trade.sql:77-79)과 락스텝, UI가 서버보다 먼저 잠겨 fail-closed. "가짜 라인이면 잠금"(오즈 불변식)의 시세판.
- 지정가 충돌 방지 collar 이식: 경고 3%/차단 5% (crypto-live.html:5875-5876).

### 5-5. 세션 가드 (결함-로그 2026-07-15 유령세션 클래스 — 2건 모두 이식)
1. **authed 판정 = `localStorage('alpexa.me')` 직접 판독** (custId+accts 있어야 참) — `AlpexaSync.me()` 사용 금지(게스트 자동생성으로 판정 오염, 결함-로그:248; alpexa-sync.js:41-51). — sports-dashboard.html:967-968
2. **태그+세션 이중 확인**: `AlpexaSync.db.auth.getSession()` 실존까지 확인, 없으면 `sessAbsent=true` + "세션 만료" UI (태그만 있고 세션 없으면 RLS가 전 행을 숨겨 돈 기능 전멸 — 결함-로그:249). — sports-dashboard.html:975-986
3. **클라이언트는 `AlpexaSync.db` 단일 재사용** — 자체 `createClient` 금지 (세션 storage 3곳 락스텝 규칙, CLAUDE.md #5; alpexa-sync.js:24-29의 sessionStorage 세션을 그대로 공유).
4. `AlpexaSync.assertIdentity()` 호출 (표시 신원 ≠ 세션 uid 드리프트 강제 재로그인). — alpexa-sync.js:288-306
5. **부분 복제 클래스 방지**: 돈 경로를 여는 위젯마다 앱의 사전 가드 목록(halt·marks·withdrawable·2FA/allowlist 트리거·busy 가드)을 표로 만들어 전 항목 이식 확인 후 오픈 (결함-로그:243-247 불변식).

### 5-6. 표시 원칙
- 잔고·보유·스테이크는 **매 로드 서버 재조회, localStorage 저장 금지** (CLAUDE.md #5). 보유가치 = `서버 qty × pricesStore 현재가` 매 틱 재파생 (crypto-live:13826-13833과 동일 — qty는 서버 그대로, 가치만 클라 파생).
- 시세 숫자에 노이즈/사인 부여 금지 — 실피드 재판독만 (crypto-live:785-799 원칙).
