## Ⅰ. A급 — 돈/보안 (활성화 전 필수 수리)

### A-1. `__alpexaHalt` 거래 킬스위치 미이식 (전 돈 버튼)
- 앱: 백오피스 `controls.trading_halt`를 5초 폴링(crypto-live.html:14643-14648)하고 Buy/Sell 제출 첫 줄에서 차단(crypto-live.html:12071).
- 대시보드: 로드/체크 모두 없음 — tkGo(dev/crypto-dashboard.html:1656), swGo(:1732), stGo(:1865), unstake(:1811), mmGo(:1949), xfGo(:2004), wdGo(:2115) 전부 halt 무풍.
- 결과: 운영자가 거래를 정지시켜도 대시보드에서는 crypto_trade/swap/stake/send가 계속 나간다. 서버 RPC에는 halt 게이트가 없어(crypto_trade.sql:56-79에 부재) **클라 체크가 유일한 관문**.
- 수리: 앱 :14644-14648 폴러 사본 + 모든 submit 첫 줄에 `if(window.__alpexaHalt) …` (앱 :12071 문구 그대로).

### A-2. 딜링 마크업(pricing_marks) 미적재 → 수수료 항상 0 + settlements 미기록
- 앱: `pricing_marks(symbol,markup_pts)`를 5초 폴링해 `window.__alpexaCryptoMarks` 적재(crypto-live.html:14636-14642) → buy/sell/redeem/swap/liquidate가 p_markup으로 전달(:13983,:14046,:14120,:14199,:14237), 수수료 발생 시 `settlements`에 crypto_fee 기록(:13992,:14056,:14085,:14182).
- 대시보드: 같은 전역을 **읽기만** 하고(dev/crypto-dashboard.html:1611, :1705) 어디서도 적재하지 않음 → 항상 0% → 대시보드 체결은 하우스 스프레드 0 + settlements P&L 행 0.
- CLAUDE.md "모든 상품에 딜링 스프레드(무스프레드 상품 없음)" 위반 — 앱과 대시보드가 같은 코인을 다른 가격에 체결.
- 수리: pullCryptoMarks 사본(:14638-14642) + 체결 성공 시 `__alpexaSettle` 계약(:14623-14625) 이식.

### A-3. `assertIdentity()` 신원 소유 게이트 미호출
- 앱: 로드 게이트 5번째로 호출(crypto-live.html:183) — 세션 uid가 표시 계정(alpexa.me)을 소유하는지 확인, 불일치 확정 시 login.html?switch=1(alpexa-sync.js:288-306).
- 대시보드: walSession(dev/crypto-dashboard.html:1036-1044)은 "세션 실존 + me 태그"만. 세션↔me 불일치(한 브라우저 계정 전환 잔재) 시 **A의 세션으로 B의 acct_no를 표시·요청** 가능. 특히 pushRequest는 cust_id/acct를 localStorage me에서 취함(alpexa-sync.js:64-71) → 입금/출금 요청이 잘못된 고객 명의로 생성될 수 있음(결함-로그 2026-07-15 유령세션 클래스의 잔여 절반).
- 참고: startIdleLogout은 alpexa-sync.js:385가 자동 실행이라 대시보드에 이미 적용 ✓ (이 건은 통과).
- 수리: 부트 시 `AlpexaSync.assertIdentity()` 1줄 (앱 :183 그대로).

### A-4. Transak 카드입금 계약 부분 복제 — 플랫폼 수취주소·대사 기록 누락
- 앱(crypto-live.html:5205-5221): body에 `walletAddress`(플랫폼 USDT 주소 0x6B1c…), `cryptoCurrencyCode:'USDT'`, `network:'ethereum'`, `fiatCurrency`, `partnerOrderId`, `partnerCustomerId:custId`, `referrerDomain` 전송 + **위젯 열기 전 pending deposit pushRequest 기록**(:5220, `from_label:'Transak '+oid`)으로 백오피스가 크레딧을 대사.
- 대시보드(dev/crypto-dashboard.html:2052-2053): `{amount, email, partnerCustomerId:acct}`만. Edge는 walletAddress 없으면 위젯 파라미터에서 생략(transak-widget/index.ts:53,:73) → **Transak이 사용자 본인 지갑주소를 받게 되어 입금이 플랫폼에 안 들어올 수 있음**. partnerCustomerId도 custId가 아닌 acct_no라 매칭 규약 상이. pending deposit 기록도 없어 백오피스가 이 주문을 기대하지 못함.
- 수리: 앱 :5205-5213 body 전체 + :5220 pushRequest를 그대로 이식.

## Ⅱ. B급 — 기능 누락/다름

### B-1. 법정화폐 출금 레일(card/ACH/wire) 통째 미이식
- 앱 Withdraw(crypto-live.html:4536-4700)는 ① 3서버(crypto/fx/sports) 소스 선택+withdrawable_for 각각(:4549-4561) ② 카드=MoonPay 오프램프(:4592-4593) ③ ACH/와이어 은행필드 검증(achValid/wireValid :4577-4578, destValid :4689) ④ 수수료 fees.withdraw*(:4594-4595) ⑤ review에서 항상 TOTP 2FA(:4893).
- 대시보드 Withdraw 팬은 **온체인 주소 출금만**(dev/crypto-dashboard.html:880-894) — 은행 출금 자체가 없음. 기획 §7 row 11 "✓"는 온체인 한정의 과대평가.
- 부수: 대시보드는 앱에 없는 BTC/ETH 온체인 출금 옵션을 신설(:881-882, network 'Bitcoin' :2148) — 앱 외부송금은 항상 asset별 `network:'ERC-20'`(crypto-live.html:13675). 승인 차감 함수 `apply_crypto_withdraw_holding`은 repo에 없어 BTC 자산 차감 계약 미검증 → **기획 §1-2 "새 돈 경로 0개" 위반 소지**. 검증 전까지 USDT만 남기는 것을 권고.

### B-2. 와이어 입금 — 회사 은행좌표·매칭 참조 미표시/미기록
- 앱: WIRE_INFO(은행/수취인/계좌/ABA/SWIFT/주소 + 고객별 참조, crypto-live.html:5106-5116, 표시 :5701-5711) + 요청에 `address:'ALPX-<custId>'`·`from_label:'Ref: …'` 기록(:13462-13469)으로 백오피스가 입금 전신을 매칭.
- 대시보드: 금액 입력→제출뿐(dev/crypto-dashboard.html:2059-2062, from:'Wire transfer') — **사용자가 어디로 송금할지 알 수 없고**, 참조 부재로 백오피스 대사 불가.

### B-3. Send/Transfer 검토(Review) 단계 부재 — 원클릭 즉시 이체
- 앱: Send는 SendReview(수수료·합계·영수증, crypto-live.html:2435) 후 confirm, Transfer도 review 경유. 잘못 이체 방지 관문.
- 대시보드: mmGo(dev/crypto-dashboard.html:1949-1970)·xfGo(:2004-2028)·swGo(:1732)·stGo(:1865)가 버튼 1클릭에 RPC 즉시 발사(unstake만 확인 스트립 있음 :1792-1798). 기획 §7 "22·25·27 인라인 확인 스트립 ✓(기능 동등)"은 **과대평가** — Send/Transfer에는 확인 스트립이 없다. 특히 Send는 되돌릴 수 없는 P2P.
- 수리: unstake식 확인 스트립을 Send/Transfer(최소)로 확대.

### B-4. Activity 원장 병합 부분 — reward/send/receive/redeem 및 앱 이력 미러 누락
- 앱: localStorage tx + 서버 `activity` 테이블 병합(crypto-live.html:13584-13613), TYPE_META 12종(:9970-9983 — reward 'Staking reward', send/receive/redeem 포함). 스테이킹 이자는 stake-accrue가 `activity`에 `stkrwd-` 기록(스펙 §4.13).
- 대시보드: crypto_trades+requests만(dev/crypto-dashboard.html:1499-1503) → **스테이킹 리워드·P2P 송수신·환매가 이력에 안 보임**. 역방향도 결손: 대시보드는 pushTx/logActivity 미러(앱 :13344-13355)를 안 하므로 **대시보드에서 한 거래가 앱 Activity에 안 나타남**(크로스디바이스 이력 계약 단절). crypto_transfers 미조회는 주석으로 정직 기록됨(:1457) — 단 activity 테이블 병합으로 보완 가능.
- 수리: `activity` 테이블 SELECT 병합 + 거래 성공 시 `AlpexaSync.logActivity` 호출.

### B-5. ALPXS 도달 경로 단절 — Redeem(#15) 사실상 실행 불가
- 서버는 ALPXS를 매길 수 있음(stock-prices/index.ts:70-74가 prices에 ALPXS 기록) → crypto_trade 체결 가능. 그러나 대시보드에서:
  - Portfolio ALPXS 행 클릭이 `if(MK_BSYM[s])` 필터로 차트/트레이드 라우팅 제외(dev/crypto-dashboard.html:1449 — ALPXS는 MK_BSYM에 없음 :1140-1142).
  - 기본 워치리스트에 ALPXS 없음(:1149) → Trade 위젯에서 ALPXS를 선택할 유일한 길이 "워치리스트에 수동 추가 후 행 클릭".
  - 차트는 'No chart feed for ALPXS'(:1262).
- 기획 §7 row 15 "✓ crypto_trade sell(서버가)"는 **RPC만 보고 UI 도달성을 안 본 과대평가**. 수리: Portfolio 클릭 라우팅에서 MK_BSYM 조건 제거(ALPXS 허용) + Trade 티켓이 alpxsPrice 사용.

### B-6. ALPXS 스테이킹 1y 전용 규칙 미강제
- 앱: `allowedPeriods = a==='ALPXS' ? ['1y'] : [flexible,90d,1y]` + 자산 전환 시 스냅(crypto-live.html:3909,:3914-3917).
- 대시보드: ALPXS 선택 시에도 Flexible/90d 칩 선택 가능(dev/crypto-dashboard.html:746-749, stCalc :1835-1855에 제한 없음) → 앱에서 불가능한 "ALPXS flexible 7%" 스테이크가 생성됨(서버 stake_crypto는 기간 제한 없음 — crypto_stake_multiterm.sql:39). 상품 규칙 불일치.

### B-7. 가격 신선도 게이트·지연 뱃지 미구현 + feedDot 상시 초록
- 기획 §1-5가 명시한 가드("서버 120초와 락스텝 거래 잠금, 뱃지 15초")가 코드에 없음: mk.q에 수신 타임스탬프 자체가 없고, 헤더 상태점은 `feedDot.classList.add('ok')` 하드코드(dev/crypto-dashboard.html:965).
- 돈은 서버 120s 게이트(crypto_trade.sql:77-79)가 막지만(fail-closed ✓), UI는 죽은 피드를 라이브처럼 표시하고 사용자는 서버 거절로만 알게 됨. WS 단절+폴백 실패 시 마지막 가격이 무기한 잔류.
- 수리: mkTouch에 ts 기록 → 15s 초과 시 뱃지/dot 회색, 120s 초과 시 티켓 잠금.

### B-8. `sync_crypto_balance` 미호출 — accounts.balance(crypto) 드리프트
- 앱: 히어로 값 변동 시 3초 스로틀 호출(crypto-live.html:1799) → 서버가 holdings×prices(+stakes)로 재계산(accounts_lock.sql:16-34).
- 대시보드: 호출 0회 → 대시보드만 쓰는 고객의 crypto 잔고가 백오피스/타 앱 표시에서 낡은 값으로 남음.

### B-9. Realtime 구독 부재 (기획 §3 자기 약속 미이행)
- 앱: crypto_holdings/crypto_stakes `postgres_changes` 구독 + 8초 폴(crypto-live.html:13851-13854, :13929). 기획 §3은 "Supabase Realtime prices 구독(신규 — 앱보다 빠름)"까지 약속.
- 대시보드: Realtime 0건 — holdings 30초 폴(dev/crypto-dashboard.html:1453), stakes 30초(:1887), activity 30초(:1531), prices 8초(:1214)뿐. 다른 기기 거래·백오피스 조정 반영이 최대 30초 지연.

### B-10. 백오피스 commands 실행기(runCmds) 미이식
- 앱: `commands`(liquidate/set_balance) 4초 폴링 실행(crypto-live.html:14658-14670). 클라 실행형 운영 도구라, 고객이 대시보드만 열어두면 백오피스 청산 명령이 영원히 pending.
- 수리(권고): 대시보드도 폴링하되 liquidate는 crypto_trade sell p_all로 실행(클라 로컬 경로 금지 원칙 유지).

### B-11. 지정가/스탑 생성 미이식 (표시·취소만)
- 앱: Limit/Stop 큐잉 + 트리거 티어 가드(wrong-side/warn>3%/too-far>5%, crypto-live.html:12053-12066, TRIGGER_* :5875-5876). 대시보드 티켓은 "Limit — soon" 비활성(dev/crypto-dashboard.html:812-813), W8은 앱이 만든 주문의 표시/취소만(:2195-2225).
- 체결기 부재(스펙 §3.4)라 위험은 없고 'soon' 표기는 정직 — 다만 기획 §7 감사표에 이 미이식이 항목으로 기재되지 않아 여기 기록.
