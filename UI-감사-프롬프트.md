# UI 기능 감사 프롬프트 — "보이는데 작동 안 하는" 것 잡기

> 사용자가 클릭하다 찾는 버그(죽은 버튼·데모 데이터·미완성 기능)를 **내가 먼저** 잡는 절차.
> 두 층으로 막는다: **(A) 정적 스캐너**(자동) + **(B) 추적 감사**(내가 화면마다).

---

## A. 자동 — `node tests/diagnose-ui.js` (verify.js에 포함)
정적으로 잡는 클래스:
- **`UI-demo-balance`** — 하드코딩 잔고(`balance/balanceUsd/mine: <숫자>`) = 진짜처럼 보이는 데모. (SEND_ASSETS $21,903 BTC가 이거였음 — 결함 #18)
- **`UI-stub-marker`** — "demo only / coming soon / not implemented / TODO" 마커.

발견 = **검토 지점**(하드 에러 아님). 각각 → **고치거나**(실데이터 읽게/완성) **검토된 안전이면 ACCEPTED 등록**.

---

## B. 추적 감사 — 정적이 못 잡는 것 (내가 화면마다 돈다)
정적 스캔으로 표현 못 하는 "기능 미연결"은 **추적**으로 잡는다. 화면 하나씩:

### B1. 모든 인터랙티브 요소 (버튼·토글·입력)
각 요소의 핸들러를 따라가서 묻는다:
- [ ] 클릭/토글하면 **실제로 뭔가 일어나나?** (서버 RPC 호출 / 실상태 변경)
- [ ] 아니면 **저장만 하고 아무도 안 읽나?** → **죽은 토글**
  - 예: `settings.notifications.liveStart` 저장됐는데 `notify(...,'liveStart')` 호출 0개였음 → 죽은 토글(고침).
  - 찾는 법: 설정 키를 grep → **쓰는 곳(set)** 과 **읽는 곳(조건/소비)** 둘 다 있나? 읽는 곳 0이면 죽음.
- [ ] 핸들러가 **빈 함수/no-op** 인가? (`onClick={()=>{}}`)

### B2. 모든 표시 숫자 (잔고·가격·보유·이자)
각 숫자의 출처를 따라가서:
- [ ] **서버 실데이터**(balances/crypto_holdings/prices/RPC)에서 오나?
- [ ] 아니면 **하드코딩/seed/추측값**인가? → 데모 누출 (#5)
  - 예: ALPXS "+2.00%"가 고정 `chg=2.0`였음. SEND BTC $21,903이 `SEND_ASSETS` 데모였음.
  - 찾는 법: 숫자 라벨 → 그 값을 만드는 변수 → seed 상수냐 서버 fetch냐.

### B3. 모든 "기능" 섹션 (입금·출금·스왑·스테이킹·송금)
- [ ] **끝까지 연결됐나?** UI → 서버 RPC → 결과 표시, 한 바퀴.
- [ ] 중간이 **시뮬레이션/스텁**인가? ("In production this routes through Plaid…" 류)

### B4. 빈 상태 / 로딩
- [ ] 데이터 없을 때(신규계정) **가짜 기본값**이 뜨나, 아니면 0/빈칸인가?
  - 데모 fallback 금지 — 안 로드됐으면 `0` 또는 `···`.

---

## 발견하면 (Control 루프)
1. **데이터 누출**(데모를 진짜처럼) → 실데이터 읽게 고치고 `*.test.js` 또는 `diagnose-ui.js` 체크 추가.
2. **죽은 토글/버튼** → 연결하거나(기능 완성) 숨김. 행위면 회귀테스트.
3. **미완성 스텁** → 완성 일정 잡거나 "준비중" 명시(가짜로 작동하는 척 금지).
4. `결함-로그.md`에 한 줄 + `node tests/verify.js` 🟢 확인.

> 핵심: **사용자가 클릭하다 찾기 전에 내가 이 두 층으로 먼저 잡는다.**
