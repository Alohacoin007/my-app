# 결함 등록부 (Defect Register) — 버그 클래스 → 근본원인 → 재발방지

> Six Sigma 정신: **결함을 "측정"하고, 근본원인을 없애고, 재발을 "도구로" 막는다.**
> 새 버그를 칠 때마다 여기에 한 줄 추가하고, 가능하면 `tests/diagnose.js`에 체크를
> 넣는다 (정밀하게 — 오탐 내는 체크는 무시당해서 더 해롭다).
>
> **게이트:** `node tests/verify.js` = 인라인 파싱 + `tests/*.test.js` + `diagnose.js`.
> 🟢 아니면 "완료" 금지.

## 근본원인 공통분모 (진짜 적)
1. **"한 곳만 바뀌겠지" 가정** → 항상 **모든 읽기·쓰기 지점 grep** 먼저 (#2, #3, #4).
2. **추측을 증거로 착각** → 실행/쿼리로 **눈으로 확인** (#5, #6, 그리고 $40 "차이"가 실은 정상이었던 건).
3. **강제 안 되는 규칙** → 도구(게이트)로 강제 (#7).

### #33 정산 고아 — settle이 "오늘 스코어보드"만 읽어 늦은 경기 영영 미정산 (2026-06-30)

## 크립토 하이브리드 스프레드 (floor + 실제 거래소 스프레드 passthrough) (2026-07-05)
- **개선(MT5 구조):** 크립토 스프레드를 고정 bps에서 **하이브리드**로 — `greatest(house floor, 실제 거래소 스프레드 bps)`. 평시엔 floor(크립토 10·주식 8·지수 6bps), 변동성/저유동 페어에서 거래소 스프레드가 floor 넘으면 자동 확대. FX는 그대로 실시간 pip 스프레드.
- **구현:** `crypto-prices` Edge를 Binance `bookTicker`로 전환(실제 bid/ask) → mid=(bid+ask)/2, `spr_pts`=실제 스프레드 **bps**. 서버 `fx_close.sql` 비FX 브랜치 `greatest(floor, spr_pts)` ↔ 클라 `fxHalfSpread` `Math.max(floorBps, feedBps)` **락스텝**. 주식(Finnhub `/quote`는 bid/ask 없음)·지수는 spr_pts=0 → floor 유지. 증명: `tests/fx-floating-spread.test.js`(floor 지배 + 30bps passthrough + client==server).
- **주의:** 크립토 `spr_pts`는 **bps** 단위(FX는 pips) — 같은 컬럼 클래스별 다른 단위, fx_close가 cls로 분기. 크립토 raw 스프레드는 평시 ≈0이라 사실상 floor(리테일 CFD 표준), 급변 시에만 확대.

- **수정:** `sports-games`의 팀 매칭을 nick(마지막단어)→**teamMatch(토큰 부분집합)**: 클럽 접미사 제거·유의미 토큰(len>2)·짧은이름 토큰이 긴이름에 전부 포함되면 매칭. 2군데 다 교체 — 이벤트 찾기(`overlayRealOdds`, **유일매칭+커밋시각 6h 이내** 아니면 잠금 유지) + 결과→팀 매칭(`oddsToCore`의 bestOutcome h2h/spreads). US 스포츠는 그대로("Reds"⊆"Cincinnati Reds"), 크로스매칭 방지("Red Sox"⊄"Chicago White Sox", NY Yankees≠NY Mets). 증명: `tests/odds-match.test.js` + 라이브 시뮬(회수 4·오매칭 0·애매0).
- **불변식:** 서로 다른 피드의 경기를 결합할 땐 마지막 단어가 아니라 **정규화 토큰 부분집합 + 유일매칭 + 시각근접**으로. 애매하면(2+ 매칭) 붙이지 말고 잠금(잘못된 배당 첨부 금지). 근본적으론 일정+배당을 **공유 이벤트 ID 한 소스**로 받으면 이 결합 자체가 사라짐(설계 B안).
- **클래스: 클라 DB SELECT가 실제 테이블에 없는 컬럼을 요청 → PostgREST가 쿼리 전체를 거절 → UI가 조용히 빈 목록.** webtrade `loadPos`가 `positions.select('...,stake,...')` — 실 테이블엔 `stake` 컬럼 없음(메인 앱은 `size`만). 데모(WT_DEMO=true)일 땐 이 쿼리가 안 돌아 숨어 있다가, 실거래(WT_DEMO=false)로 켜고 실계정 로그인·주문 후 처음 실행되며 발현: 서버엔 포지션이 멀쩡히 들어갔는데(모바일엔 보임) webtrade만 "No open positions". 하네스는 실 세션·DB·CDN이 없어 이 실쿼리를 못 돌려 파싱/로직/렌더 테스트를 전부 통과함(사각지대).
- **수정:** `loadPos` SELECT를 trading.html의 검증된 쿼리와 정확히 일치(`local_id,symbol,side,size,open_price,pnl,status`)로 교체. `stake`·`created_at`(positions엔 미증명) 제거.
- **불변식/포카요케:** 클라가 `positions`/`accounts`에서 SELECT하는 모든 컬럼은 **증명된 실스키마 집합**(fx_open INSERT + trading.html 실셀렉트) 안이어야 함. `tests/webtrade-db-columns.test.js`가 정적으로 강제(실 DB 없이도 유령 컬럼 차단). 근본적으론 스키마-클라 컬럼을 한 소스에서 생성하면 어긋남 자체가 사라짐.
