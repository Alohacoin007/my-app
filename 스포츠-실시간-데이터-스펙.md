**배당 최악 지연(라이브 경기)** ≈ A(≤60s) + B(≤60s+실행지터) + C(≤1s) + D(≤30s) ≈ **150초**; 평균 ≈ 75초.
**라이브 스코어 지연** ≈ D′ 9초가 지배 (ESPN 직결이 B의 1분 크론을 우회).
**프리매치 배당**은 A가 9분 모드라 최악 ≈ 9분 + 90초 — 단 프리매치 라인은 원래 느리게 움직이므로 설계상 허용.

> **병목 결론: 지배 홉은 서버 크론 2단(A+B ≈ 120초)이다.** 클라 폴 30초(D)는 2위, feed 캐시 1초(C)는 무시 가능. 즉 대시보드가 앱보다 빨라질 수 있는 여지는 D를 푸시로 바꾸는 것(30s→~0)까지고, 그 이상은 서버 크론 튜닝 영역(이 스펙 범위 밖). **대시보드의 진짜 목표는 "앱과 동일 순간"이지 "앱보다 빠름"이 아니다** — 같은 feed Edge, 같은 주기 상수를 쓰면 자동 달성.

## 2. 대시보드 데이터 레이어 설계 — 스토어 4개, 구독 API 하나

원칙: **webtrade priceStore 패턴 이식** — 페이지 전체가 스토어에 **한 번** 구독하고(webtrade.html:688, :1067), 위젯은 `subscribe(fn)`으로만 받는다. 파서는 하나(_apply), notify는 400ms 스로틀(webtrade.html:1070). 위젯별 개별 폴링/개별 채널 금지.

### 2.1 gamesStore (경기+배당 — 읽기 전용 브로드캐스트)

```
gamesStore = {
  games:[], ts:0, live:false, seq:0, listeners:new Set(),

  // 소스 3종 + 우선순위 게이트 (webtrade _apply의 src 게이트 이식 — webtrade.html:990-1006)
  //  'feed' : 전체 스냅샷 (feed Edge → lgToGame 변환) — 배당의 유일 소스
  //  'wake' : live_games Realtime UPDATE → 페이로드는 버리고 feed 재fetch 트리거만 (§4 SQL 필요; 없어도 동작)
  //  'espn' : watchLiveScores 9s — 스코어/time/live/suspendUntil만 패치, 배당 배열 절대 불변
  _applySnapshot(raw, fetchSeq),   // 단조 게이트: fetchSeq < this.seq → 폐기 (늦게 도착한 옛 스냅샷이 새것을 되돌리지 못하게)
  _patchScores(fresh),             // 필드 화이트리스트: home.sc/away.sc/time/state/live + 점수변동 서스펜드(15s)만

  start(){ this._startRealtimeWake(); this.pull(); setInterval(()=>!document.hidden&&this.pull(), 30000);
           setInterval(()=>!document.hidden&&this.pullScores(), 9000); },
  subscribe(fn){...}, get(gid){...},
}
```

- **읽기 경로 = 앱과 동일**: `functions.invoke('feed',{body:{t:'games'}})` → 실패 시 `from('live_games').select('data').eq('id','all')` 폴백 (sports-live.html:3621-3637과 동일). 변환도 동일 lgToGame 계약(스포츠-앱-문법-스펙.md §5.1).
- **폴 주기 = 앱과 동일 30s/9s.** 소스(B홉)가 60초인데 클라만 더 조여봐야 DB만 두들긴다(feed 캐시가 흡수하지만 이득 0). 하네스도 sub-1s 폴링을 금지(tests/sports-dashboard-fidelity.test.js의 `setInterval(...,<1000)` ban).
- **Realtime wake(선택, §4 SQL 적용 시)**: `channel('lg-wake').on('postgres_changes',{table:'live_games'},()=>this.pull())`. **페이로드를 파싱하지 않는 이유** 2가지: ① live_games.data는 경기 전체 배열(수백 KB 가능) — Realtime 페이로드 한도 초과 시 이벤트 유실 위험 [ESTIMATE: Supabase 기본 한도 ~1MiB]; ② 파싱 경로 이원화 금지 — 웹트레이드가 "one parser, one pipeline"으로 단위 드리프트 재발을 막은 원칙(webtrade.html:1031-1034) 그대로. wake는 "새 데이터 있음" 신호일 뿐, 진실은 항상 feed 경유. publication 미적용 상태에서도 구독 에러는 무해(이벤트만 안 옴, 30s 폴이 그대로 커버 — 웹트레이드 주석 :1033-1034와 동일한 하위호환).
- **우선순위 게이트**: webtrade는 "fresh(<10s) WS 틱을 3s 크론이 뒤로 끌지 못하게" src·타임스탬프로 게이트한다(webtrade.html:995-999). 스포츠 이식판은 ① 스냅샷 간 단조 seq(비순서 도착 방지), ② espn 패치의 **필드 화이트리스트**(빠른 스코어가 느린 배당을 절대 못 건드림 — 반대로 60초 배당 스냅샷이 9초 스코어를 되돌리는 건 무해: 다음 9초 틱이 재패치하고, 점수변동 서스펜드는 `_suspend` 맵이 스냅샷 교체를 넘어 생존 — sports-live.html:3570, :3577-3588 계약 유지).

### 2.2 betsStore (open/settled 베팅)

- **소스**: 앱 §3.8 계약 그대로 — open=`positions`(cust_id·server='sports'·kind='bet'·status='open') rebuild, settled=`settlements`(bet_won/bet_lost) rebuild (sports-live.html:2314-2334, :2406-2424).
- **푸시**: `channel('sportsbet-'+acct).on('postgres_changes',{table:'positions',filter:'acct_no=eq.'+acct}, wake)` — 앱 :2359-2361 복제. DELETE(=정산 선점·캐시아웃)도 REPLICA IDENTITY FULL 덕에 filter를 통과해 즉시 도착.
- **폴백**: 6초(앱 :2350와 동일 상수). wake와 폴 모두 같은 rebuild 함수 하나로 수렴(이중 파서 금지).
- Realtime은 RLS를 따르므로 **로그인 세션의 supabase 클라이언트**로 구독(대시보드도 alpexa-sync.js 공유 — sessionStorage 세션 계약은 CLAUDE.md #5 참조).

### 2.3 balanceStore (잔고 — 표시 전용)

- 앱 계약 100% 복제: `let balance=0`(로컬 시드 금지) → `pullBalances()`(alpexa-sync.js:219-226) 로드 시+10s+visibilitychange (sports-live.html:3310, :3325-3326) + `channel('sportsbal-'+acct)` accounts Realtime wake (:3334-3336) + **7초 로컬 변경 가드**(:3301-3303) + 미로드 `'···'` 표기(:2599-2607, 문법-스펙 §4.2).
- 대시보드 위젯 전부(헤더 잔고·Pending·P&L 카드)가 이 스토어 하나에서 읽는다. Pending = `openBets.reduce(Σstake)` — betsStore 파생값이지 별도 폴링 아님 (앱 pendingTotal :2537).

### 2.4 controlsStore (운영 파라미터)

- 앱은 4개 폴(5s×4: pricing margin :3776 · halt :3782 · liveBetting :3788 · ctrl 전체 :3795)을 따로 돈다. 대시보드는 **`from('controls').select('key,val')` 1쿼리 + `from('pricing')` 1쿼리, 5초**로 통합해 같은 전역(`__alpexaSportsMargin/__alpexaHalt/__alpexaLiveBetting/__alpexaCtrl`)을 채운다 — 값·주기 동일, 쿼리 수만 4→2. (앱 전역명을 그대로 쓰는 이유: 문법-스펙 부록의 공유 런타임 파라미터 계약 + fidelity 핀.)

### 2.5 구독 API (전 위젯 공통)

```
un = gamesStore.subscribe(fn)   // fn(games) — 400ms 스로틀 notify (webtrade.html:1070 패턴)
un = betsStore.subscribe(fn)    // fn({open, settled})
un = balanceStore.subscribe(fn) // fn({balance, loaded, pending})
```
위젯은 스토어 밖에서 fetch/채널을 만들지 않는다. 연결상태 LED는 `gamesStore.live`(마지막 feed 성공 여부) — webtrade FeedDot 패턴(webtrade.html:2625-2626).

## 3. sports_odds를 직접 읽지 않는다

- 클라 주경로의 배당은 이미 `sports-games`가 서버에서 오버레이해 live_games에 구워 넣는다(overlayRealOdds — sports-games/index.ts:88-92, :249-251). 앱이 `sports_odds`를 직접 읽는 곳(applyRealOdds — sports-live.html:3518, :3546)은 **ESPN 직접 폴백 경로에서만** 호출된다(:3658).
- 대시보드는 이 폴백까지 복제할 필요 없음 — live_games가 비면 "unavailable" 상태를 그대로 보이는 게 낫다(폴백 이원화 = 앱과 다른 데이터를 보는 창구). 따라서 sports_odds publication 추가도 불필요.

## 4. 필요한 서버 변경 — 검토된 SQL 초안 (실행은 사용자, 선택 사항)

### 4.0 (권장) feed Edge 배포 — 코드는 있는데 서버에 없음
실측(§1.1 C): `feed`가 404 NOT_FOUND = **미배포**. 지금은 모든 클라가 live_games를 직접 REST로 읽는다(1인당 30초 1회라 당장 무해하나, 유저 N명 = DB 읽기 N배). 대시보드가 유저를 늘리기 전에 배포 권장:
```
supabase functions deploy feed        # (사용자 실행 — JWT verify 기본값 유지 OK: 함수가 anon 키로 동작)
```
배포 전이어도 대시보드는 앱과 동일한 "feed 우선 → live_games 직접 폴백" 코드로 그대로 동작한다(§2.1) — 배포 순서 자유.

### 4.1 (선택) live_games를 Realtime publication에 추가 — 클라 D홉 30s 폴 지연을 wake로 ~0초화. 지배 홉(서버 크론 120s)은 그대로이므로 효과는 "최악 150s→120s". 안 해도 대시보드는 30s 폴로 앱과 동일하게 동작한다.

```sql
-- live_games를 supabase_realtime publication에 추가 (멱등 — 리마인드-리스트.md:244의
-- crypto_holdings 추가와 동일한 DO 블록 패턴)
do $$ begin
  if not exists (
    select 1 from pg_publication_tables
     where pubname='supabase_realtime' and schemaname='public' and tablename='live_games'
  ) then
    alter publication supabase_realtime add table public.live_games;
  end if;
end $$;
-- 확인: select tablename from pg_publication_tables where pubname='supabase_realtime';
-- 롤백: alter publication supabase_realtime drop table public.live_games;
```

주의사항 (READ-BEFORE-WRITE 체크 완료):
- live_games 쓰기는 sports-games Edge(service role) 단독·1분 1회 upsert(index.ts:253-259)라 이벤트 폭주 없음. RLS SELECT는 `using(true)` 공개(live_games.sql:15-16)라 anon 구독도 이벤트 수신 가능.
- 클라는 이벤트 **페이로드를 쓰지 말고 wake로만** 사용(§2.1 — 페이로드 한도 [ESTIMATE], one-parser 원칙).
- 이 SQL 없이도 대시보드 코드가 그대로 동작해야 함(구독 실패 무해 — webtrade.html:1033-1034 하위호환 원칙). **배포 순서 자유.**

그 외 서버 변경 **불필요**: accounts/positions/settlements publication은 이미 켜져 있고(§1.3), feed Edge·크론은 앱과 공용.

## 5. 신선도 게이트 — 피드 끊김 시 표시/베팅 차단

앱 계약(무관용, 동일 상수):
- `__alpexaOddsTs` = 마지막 성공 스냅샷 시각 — 성공 apply에서만 갱신 (sports-live.html:3641, :3658)
- `ODDS_STALE_MS = 90000` · 체크 3초 (`refreshOddsStale` — :3568, :3573-3574). **90초를 줄이지 말 것**: 소스 자체가 60초 주기라 여유는 ~30초뿐 — 더 조이면 정상 상태에서 거짓 스테일.
- `__alpexaOddsStale === true` → liveOK false = 전 라이브 베팅 잠금 + `'Odds updating — try again'` (:1717, :1723)
- 점수 변동 → 해당 경기 `SUSPEND_MS = 15000` 잠금 (:3567, :3583)

대시보드 추가 규칙:
1. **표시**: 스테일 시 배당 셀을 dim/🔒 처리 + 헤더에 "Odds updating" 배너. 마지막 값은 계속 보여준다(빈 화면 금지 — 웹트레이드가 dropout에 자기 스프레드 유지하듯 :1001-1003) — 단 **베팅 진입만 차단**.
2. **베팅**: liveOK 3조건(문법-스펙 §1.5: `__alpexaLiveBetting` && !suspendUntil && !`__alpexaOddsStale`) 통과 없이는 슬립 추가 자체를 막는다. 서버 place_bet re-price가 최종 관문(place_bet_server_odds.sql:83-86)이지만 **UI가 열려 있으면 그 자체가 결함**(문법-스펙 §6 #11).
3. **연결 LED**: `gamesStore.live=false`(feed 연속 실패) → LED 회색 + "reconnecting". Realtime CHANNEL_ERROR/TIMED_OUT은 무시하고 폴이 커버(푸시는 보너스, 진실은 폴).
4. **탭 복귀**: `visibilitychange` → 즉시 pull 3종(games/bets/balance) — hidden 동안 폴이 멈추므로(앱 전 폴링의 `!document.hidden` 가드와 동일) 복귀 순간 스테일 화면을 최속 해소.

## 6. 금지 목록 — 데이터 레이어가 하면 안 되는 것

문법-스펙 §6의 13종(클라 ledger/positions INSERT, 로컬 잔고, 가짜 배당, 클라 정산, 프롭 제출 등)은 전부 그대로 적용. 데이터 레이어 고유 추가분:

| # | 금지 | 이유/근거 |
|---|------|-----------|
| D1 | **자체 집계 잔고** — ledger를 클라에서 Σ해서 balance 계산, 또는 베팅/정산 이벤트로 balance를 델타 갱신 | 잔고는 `accounts.balance` 단일 진실(트리거 trg_apply_ledger — CLAUDE.md 돈 불변식). 델타 추측 = $60/$80 이중차감 사고 클래스 |
| D2 | **클라 쓰기 전면 금지** — live_games/prices/sports_odds/controls/pricing에 insert/update/upsert | 전부 Edge(service role) 소유. live_games는 공개 write 정책 자체가 없음(live_games.sql:12-14) |
| D3 | **1초 미만 폴링 / 위젯별 중복 폴링** — 스토어 밖 setInterval fetch | 소스가 60초인데 이득 0, DB/feed만 두들김. 하네스가 정적 차단(fidelity test `setInterval(...,<1000)` ban). 페이지당 채널 수도 스토어당 1개로 고정 |
| D4 | **Realtime 페이로드를 진실로 파싱** (live_games row.data 직접 채택) | 파서 이원화 + 페이로드 한도 유실 위험 — wake 전용 (§2.1, webtrade.html:1031-1034 원칙) |
| D5 | **espn 스코어 패치가 배당/oddsReal 필드를 건드림** | 빠른 소스가 느린 소스의 권위 필드를 오염 — 필드 화이트리스트 강제 (§2.1 게이트) |
| D6 | **신선도 상수 독자값** (90s/15s/3s/30s/9s/6s/10s 변경) | 앱과 다른 순간에 잠기고 풀림 = "같은 순간, 같은 데이터" 위반 + 아비트라지 창 |
| D7 | **sports_odds 직접 읽기** | 주경로 우회 = 앱과 다른 배당을 보는 창구 (§3) |
| D8 | **돈/베팅/배당의 localStorage 캐시** (세션 토큰·UI 설정 제외) | CLAUDE.md #5 — 계정간 누수·stale·이중계산의 근원 |
| D9 | **feed 실패 시 데모/가짜 게임 생성** | 앱도 하드 실패 시 빈 목록 + 배너(sports-live.html:3668-3674). 가짜 라인 = 오즈 불변식 위반 |
| D10 | **폴링과 Realtime이 서로 다른 rebuild 경로를 타는 것** | wake·폴 모두 같은 함수 하나로 수렴 — 두 경로가 같은 일을 하게 두지 않는다(CLAUDE.md #3 "진실은 한 곳") |
