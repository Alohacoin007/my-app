# 스포츠 마스터 감사 (일일 자동) — 스포츠-마스터-감사.md

> 매일 정해진 시간(기본 **저녁 6시 라스베가스/PDT**)에 **자동으로 한 번** 돈다.
> 묻는 것: **오늘 들어온 경기 데이터(일정·시간·결과)가 정상이었나? 우리가 받은 유저 베팅이 전부 올바르게 기록·정산됐나? 돈이 새거나 두 번 지급된 게 있나?**
> 이건 코드 감사(`마스터-감사-프롬프트.md`)와 **다르다.** 그건 "코드에 버그 클래스가 있나"를 보고, 이건 **"오늘 실제 운영 데이터가 멀쩡했나"**를 본다.
>
> ⚠️ 아래 SQL은 **검토된 쿼리**다. 배포·실행은 **사용자가** 한다(Claude은 SQL/Edge 못 돌림). 처음 1회는 컬럼명을 실제 스키마와 대조하고 켠다.

## 0. 진실의 위치 (스포츠 데이터 지도)

| 무엇 | 진실 저장소 | 누가 씀 | 멱등 ref / 키 |
|---|---|---|---|
| 경기 일정·점수·라인 | `live_games`(id='all', data jsonb) · `sports_odds`(sport→jsonb) | Edge `sports-games`(1분) · `sports-odds`(스마트폴) ← ESPN(무료) + Odds API | gid = `<LG>_<espnEventId>` |
| 유저 베팅(미정산) | `positions`(status='open', meta.legs) | RPC `place_bet`(서버가 배당 재검증) | local_id |
| 스테이크 차감 | `ledger`(kind='bet', amount=-stake) | `place_bet` → 트리거 `trg_apply_ledger` | `betstake-<local_id>` |
| 결과·정산 | `settlements`(kind, pnl, detail) + `positions` **삭제로 선점** | Edge `sports-settle`(5분) ← ESPN state='post' | — |
| 당첨 지급 | `ledger`(kind='bet_won', amount=+payout) | `sports-settle` | `betpay-<local_id>` |
| 잔고(표시) | `accounts.balance` (= opening + Σledger) | 트리거만. 클라는 읽기만(`window.__srvBal`) | — |

**불변식 (이게 깨지면 빨강):**
1. `accounts.balance == opening_balance + Σ(ledger.amount)` — 항상.
2. 모든 `betpay-<id>` / `betstake-<id>` ref는 **유일** (두 번 = 이중지급/이중차감).
3. 정산된 베팅은 `positions`에서 **삭제**되고 `settlements`에 **정확히 1행**.
4. `meta.legs[].am`(체결 배당)은 **잠김** — 정산은 이걸로 계산, 라이브로 덮어쓰지 않음(#21).
5. 이긴 베팅 `pnl >= 0`, 진 베팅 `pnl = -stake`.

## 1. 매일 6시 자동 체크 (C1~C9 · 결정론적 SQL)

> 각 체크는 **빈 결과 = 🟢**. 행이 나오면 그 건수만큼 🟡/🔴. server는 항상 `'sports'`.

### C1 · 묵은 미정산 베팅 (결과가 안 들어옴) 🔴
경기가 끝났는데(또는 취소·연기) 정산이 안 돼 `open`으로 박힌 베팅. 유저 돈이 묶임.
```sql
SELECT cust_id, acct_no, local_id, stake, created_at, meta->'legs'->0->>'gid' AS gid
FROM positions
WHERE server='sports' AND status='open'
  AND created_at < now() - interval '36 hours'
ORDER BY created_at;
```
**빨강이면:** 각 gid를 ESPN에서 직접 조회 → state 확인. `post`면 정산 누락(팀이름 매칭 실패 의심 C/판정 §2), 사라졌으면 취소·연기 경기(수동 정산/환불 필요).

### C2 · 이중지급 / 이중차감 (멱등 ref 중복) 🔴🚨
같은 ref가 2번 = 돈이 두 번 움직임. **절대 나오면 안 됨.**
```sql
SELECT ref, count(*) cnt, sum(amount) total
FROM ledger
WHERE server='sports' AND ref LIKE ANY (ARRAY['betpay-%','betstake-%','xfer-%','fix-%'])
GROUP BY ref HAVING count(*) > 1;
```
**빨강이면:** 즉시 보고. 정산/캐시아웃 레이스 또는 ref 유니크 제약 누락. (구조: ref에 UNIQUE 제약이 있어야 함 — 없으면 그게 근본원인.)

### C3 · 스테이크 차감 누락 (공짜 베팅) 🔴🚨
포지션은 있는데 차감 ledger가 없음 = 유저가 돈 안 내고 베팅.
```sql
SELECT p.local_id, p.cust_id, p.stake, p.created_at
FROM positions p
WHERE p.server='sports'
  AND NOT EXISTS (SELECT 1 FROM ledger l WHERE l.ref = 'betstake-'||p.local_id);
-- 정산까지 간 것도: settlements 기준으로 한 번 더
SELECT s.local_id, s.stake
FROM settlements s
WHERE s.server='sports'
  AND NOT EXISTS (SELECT 1 FROM ledger l WHERE l.ref = 'betstake-'||s.local_id);
```
**빨강이면:** `place_bet`에서 ledger insert가 실패했는데 position insert는 성공한 것(원자성 깨짐). RPC가 한 트랜잭션인지 확인.

### C4 · 잔고 불변식 위반 🔴🚨
표시 잔고가 ledger 합과 안 맞음 = 어딘가 우회 기록.
```sql
SELECT a.acct_no, a.balance,
       a.opening_balance + coalesce(sum(l.amount),0) AS expected
FROM accounts a
LEFT JOIN ledger l ON l.acct_no = a.acct_no
WHERE a.server='sports'
GROUP BY a.acct_no, a.balance, a.opening_balance
HAVING a.balance <> a.opening_balance + coalesce(sum(l.amount),0);
```
**빨강이면:** 트리거 우회 또는 클라가 accounts/ledger 직접 건드림(RLS 구멍). 최우선.

### C5 · 정산 부호 오류 (이긴 베팅 -pnl / 진 베팅 +pnl) 🔴
```sql
SELECT local_id, kind, stake, pnl
FROM settlements
WHERE server='sports'
  AND ((kind='bet_won' AND pnl < 0) OR (kind='bet_lost' AND pnl > 0));
```
**빨강이면:** `sports-settle` gradeLeg/payout 부호 버그. 지급액 직접 재계산해 대조.

### C6 · phantom 델타 누수 (syncSportsBal) 🟡→🔴
알려진 약점(#5/#63): 클라가 잔고 델타를 추측해 ledger에 올리던 길. 정산 후 `sports_app` 류 보정행이 보이면 이중계산 신호.
```sql
SELECT cust_id, kind, count(*), sum(amount)
FROM ledger
WHERE server='sports' AND kind NOT IN ('bet','bet_won','bet_cashout','deposit','withdraw','adjust')
GROUP BY cust_id, kind
ORDER BY count(*) DESC;
```
**빨강이면:** 예상 못 한 kind가 잔고를 움직이고 있음. 그 kind를 누가 쓰는지 grep → 클라 경로면 구조적 폐쇄(#5).

### C7 · 오즈 신선도 (스테일 라인으로 베팅 위험) 🟡
```sql
SELECT sport, updated_at, now() - updated_at AS age
FROM sports_odds
WHERE updated_at < now() - interval '15 minutes';
```
**빨강이면:** `sports-odds` 크론 실패(CRON_SECRET, API 키 만료, Odds API 쿼터 소진). 유저가 오래된 라인에 베팅 → 우리 손해. 크론·쿼터 확인.

### C8 · 라이브 점수/일정 신선도 🟡
```sql
SELECT updated_at, now() - updated_at AS age
FROM live_games WHERE id='all';
```
**빨강이면(age > 5분):** `sports-games` 크론 멈춤 → 앱이 옛 일정/점수 표시.

### C9 · SGP 헤어컷 검증 (이긴 SGP만, 표본) 🟡
같은 경기 다리(SGP)는 헤어컷 0.25 적용. 지급액이 공식과 맞는지.
```sql
SELECT local_id, stake, pnl, detail
FROM settlements
WHERE server='sports' AND kind='bet_won'
ORDER BY created_at DESC LIMIT 50;
```
판정: 모든 leg `gid` 동일이면 SGP → `payout ≈ stake × (1 + (Πdec − 1) × 0.75)`. `pnl = payout − stake`와 $0.01 초과 차이면 🔴. (place_bet과 sports-settle의 `SGP_HAIRCUT`이 둘 다 0.25인지 코드 확인.)

## 2. 사람/Claude 판단 체크 (쿼리로 못 잡음 · 주간 or 빨강 난 날)

쿼리는 "숫자가 안 맞음"은 잡아도 **"왜"**는 못 잡는다. C1·C5가 빨강이면 아래로 파고든다.

- **팀이름 매칭 실패** — `place_bet`이 저장한 팀명("Los Angeles Lakers")과 ESPN 결과명("LAL"/"LA Lakers")이 `gradeLeg`의 부분일치로 안 맞으면 grade=null → 영원히 미정산(=C1의 흔한 원인). 묵은 베팅의 gid를 ESPN에서 직접 조회해 이름 대조.
- **타임존 (라스베가스/PDT)** — `live_games.iso`는 UTC. 표시·정산 모두 PDT 기준이어야 함. 미래 경기 3개의 iso를 PDT로 환산해 비상식 시간(새벽 3시 킥오프 등) 없는지.
- **취소·연기 경기** — 특히 SOC(축구) 토너먼트 종료 시 리그 path가 피드에서 빠지면 그 경기 베팅은 결과를 못 받음. 묵은 베팅의 gid prefix(예 `SOC_`)를 ESPN 해당 path에서 직접 조회.
- **gid 패턴 일관성** — `sports-games`가 만든 gid 형식이 `sports-settle`·`place_bet` 조회 형식과 **정확히 같아야** 매칭됨. LEAGUES 목록(games ↔ settle)이 어긋나면 통째로 미정산.

|---|---|---|---|
| C1 | 결과 못 받아 돈 묶임 | ESPN state / 팀이름·gid | 🟥 USER(수동 정산·환불 SQL) |
| C2 | 이중지급/이중차감 | ref UNIQUE 제약 | 🟥🚨 USER(제약 추가 + 환수) |

| C4 | 잔고 우회 | 트리거·RLS | 🟥🚨 USER(RLS 잠금) |

## 5. 어떻게 "매일 6시에 저절로" 도나 (실행 옵션)

세 가지. **추천: A** (기존 인프라와 동일 패턴, Claude 없이도 돔).

- **A. Edge Function `sports-audit` + pg_cron** — `sports-settle`와 똑같은 구조. C1~C9 SQL을 함수가 돌리고 결과를 `audit_reports` 테이블에 1행 기록(+🔴면 알림/이메일). pg_cron이 **저녁 6시 PDT = `0 1 * * *` UTC**(PDT는 UTC−7)로 호출. → 사용자가 `cron_secure.sql`에 한 줄 + 함수 배포.
- **B. `node tests/sports-daily-audit.js` + GitHub Actions cron** — 같은 SQL을 노드가 읽기전용 키로 실행, 리포트를 Actions 로그/Slack로. 리포지토리 안에서 관리됨.
- **C. 예약된 Claude 세션이 이 MD를 실행** — C1~C9 결정론 부분 + §2 판단 부분까지 Claude이 ESPN 직접 조회하며 추론. 가장 똑똑하지만 Claude 스케줄+DB 접근 필요. → A/B가 매일 돌고, **🔴 난 날만** C로 깊게.

> ⏰ **PDT 환산 주의:** pg_cron은 UTC. 6pm PDT(서머타임 −7) = **익일 01:00 UTC** → `0 1 * * *`. 겨울 PST(−8)면 `0 2 * * *`. (우리는 라스베가스 기준.)
